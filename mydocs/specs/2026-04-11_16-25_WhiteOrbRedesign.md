## 1. Requirements (Context)
- **Goal**: 重新设计白球，使其不再只是偏无聊的群伤工具，而是更贴合 Hollow Knight 实战环境，并且最好能与黄球、黑球槽形成联动。
- **In-Scope**: 梳理当前三种球的职责、白球当前机制弱点、可用联动点、支持型替代方案、落地实现入口。
- **Out-of-Scope**: 本轮不直接改代码；不先承诺最终数值；不在未完成设计评审前修改现有平衡；不再依赖原版 HK 或公开 mod 作为设计参考。

## 2. Architecture (Optional - Populated in INNOVATE)
- **当前已确认的设计约束**:
  - 白球可以从主伤害球改为辅助/节奏球。
  - 白球需要同时对黄球与黑球提供价值，但不能成为独立最优解。
  - 白球可以削弱清杂能力，换取更适合 HK 常见单 Boss / 少量精英战的实战价值。
  - 设计目标应避免“只靠白+黄”或“只靠白+黑”就压过三色混搭的情况。
  - 白球不承担“单拿也好用”的职责，它的上场理由必须来自组合收益。
  - 至少在一个常见组合窗口中，`1 白 + 2 其它` 的被动总收益必须显著高于 `3 黄`、`3 黑`、或其它不含白的三球组合之一；否则白球会变成负优化槽位。
  - 设计来源以原创性优先，不再追求与原版 HK 或公开 mod 的“可对照相似物”。
  - 需要正面解决“玩家天然偏向全黄球、忽视黑球”的结构性倾向：白球的设计不能只是奖励混搭，而必须明确提高黑球的上场价值。
  - 白球必须带有一定随机性，避免出现唯一最优的固定出球顺序；但这种随机性必须是可理解、可预期、可部分操控的，不能是纯赌脸。
  - 用户希望设计尽量简单，不要引入过多中间状态、过多预告层、或复杂的支援池系统。
  - 核心目标被进一步收敛为：通过“黑球给盾 -> 更敢受击 -> 受击推进回合并触发球被动”的链路，让带黑球的玩法能创造高于纯黄球无伤激发的总收益。
  - 用户进一步澄清：不需要对白球专门设计“耗盾受击特判”；真正需要的是让白球**普遍放大其它球的被动价值**。
  - 黑球的价值来源应是：它最容易安全地创造更多受击推进，因此最容易把“白球放大的其它被动”多次兑现出来。
  - 白球不应该直接把盾转成伤害，而应该遵循：`盾让你更敢受击 -> 受击让回合推进 -> 回合推进让被动触发 -> 被动造成收益/伤害`。

- **当前已确认的实现约束**:
  - 现有定义接口只有 `GetInitialDamage` / `OnPassive` / `OnEvocation`，白球的状态目前只通过 `OrbInstance.CurrentDamage` 持有。
  - 若白球改成辅助型，可能需要把 `CurrentDamage` 重新解释成“层数 / 充能 / 强化值”，或者扩展 `OrbInstance` 结构。
  - 白球若要联动其他球，当前最直接入口是：
    - `WhiteOrbDefinition.OnPassive(...)`
    - `WhiteOrbDefinition.OnEvocation(...)`
    - `OrbSystem.TriggerPassiveOrbs(...)`
    - `OrbTriggerContext.Runtime`
  - 如果白球状态不再能用单个整数完整表达，则必须同步扩展：
    - `src/Orbs/Runtime/OrbInstance.cs`
    - `src/Orbs/Runtime/OrbInstanceSnapshot.cs`
    - `src/Orbs/OrbPersistentState.cs`
    - `src/Orbs/Runtime/OrbRuntime.cs` 的生成 / 快照 / 恢复链路

- **当前已确认的数值原则**:
  - 白球不能通过自己直接打伤害来证明价值，价值应主要体现在“放大黄球命中质量”与“提高黑球防御转化效率”。
  - 白球的放大收益要足够明显，确保玩家愿意牺牲一个纯输出/纯防御槽位来换组合收益。
  - 白球的放大收益又不能无限叠加，否则会反过来把黄球或黑球变成白球的挂件。

- **当前已确认的创意方向原则**:
  - 白球应更像“节奏调度器 / 协同器 / 转化器”，而不是第三种传统意义上的输出球。
  - 创意优先于借鉴；评估标准是玩法辨识度、轮转张力、与黄球/黑球的互补性，而不是是否存在外部同类实现。
  - 接下来直接产出多个原创候选方案，并比较其：
    - 对黄球的放大方式
    - 对黑球的放大方式
    - 单白时的空转程度
    - `1 白 + 2 其它` 对比 `3 非白` 的收益差
  - 随机性的目标不是制造惊喜，而是破坏固定背板，让玩家根据当前球序和当前白球“预告状态”临场调整。

- **原创候选方案 A：导流白球（Conduit）**
  - **核心幻想**: 白球自己不结算收益，而是把本回合轮转“导流”给邻近两种球，让混搭时每次回合推进都更值钱。
  - **被动**: 白球每次触发时，为其右侧最近的黄球附加 `导流`，使该黄球下次被动改为 `锁定最低血敌人` 或 `追加一次半伤追击`；同时为其左侧最近的黑球附加 `凝流`，使该黑球本次或下次被动额外获得 `+1` 冰花。
  - **激发**: 被挤出时，立刻触发一次“全导流”：本轮中下一次黄球被动必定强化，下一次黑球被动额外转化冰花，并按当前白球层数提高幅度。
  - **优点**: 价值完全来自提高黄球命中质量与黑球成盾速度，天然适合 Boss 战；`1 白 + 1 黄 + 1 黑` 会比 `3 黄` 或 `3 黑` 更平衡且更稳定。
  - **风险**: 若导流可无限覆盖，会导致白球成为通用最优中枢。
  - **主要平衡旋钮**: 导流只作用于最近一颗黄球和一颗黑球，且不可叠加，只能刷新。

- **原创候选方案 B：共振白球（Resonance）**
  - **核心幻想**: 白球记录“上一颗触发的非白球类型”，在黄/黑之间制造交替奖励，鼓励双色轮转而不是堆同色。
  - **被动**: 若上一颗被动触发的是黄球，则白球给黑球挂上 `共振护层`，让下一次黑球被动获得额外冰花；若上一颗触发的是黑球，则白球给黄球挂上 `共振裂隙`，让下一次黄球被动附加一次小追击或提高倍率。
  - **激发**: 立刻结算一次“双向共振”：下一次黄球和黑球被动都得到加强，但数值低于持续轮转时的总收益。
  - **优点**: 天然反同色堆叠，越混搭越强；`1 白 + 黄 + 黑` 因为能持续交替吃奖励，会明显优于 `3 黄` 或 `3 黑` 的线性收益。
  - **风险**: 规则理解成本略高，UI/日志需要清楚显示当前共振目标。
  - **主要平衡旋钮**: 只有“类型切换”才触发强收益，连续同类只给极弱或不给收益。

- **原创候选方案 C：折射白球（Refraction）**
  - **核心幻想**: 白球把别的球本来单一路径的收益折射成另一条路径，让输出和防御互相借力。
  - **被动**: 当黄球被动命中后，白球把其中一部分收益折射成 `冰花碎片` 给黑球；当黑球被动获得冰花后，白球把其中一部分防御势能折射成 `蓄雷` 给黄球，使下一次黄球被动更强或更精准。
  - **激发**: 立即把当前战场上的“折射记录”全部兑现：最近一次黄球强化立刻生效、最近一次黑球转化立刻补足一段冰花。
  - **优点**: 最符合“转化器”定位，白球永远不自己出货，而是让黄球和黑球互相供养；混搭辨识度最强。
  - **风险**: 若转化比例太高，会让黄球和黑球都像被动吃白球红利。
  - **主要平衡旋钮**: 折射只转化“部分增量”而非全部收益，且每轮只触发一次输出转防御、一次防御转输出。

- **原创候选方案 D：押后白球（Delay）**
  - **核心幻想**: 白球不直接增强当下，而是把收益延后结算，制造“存一拍、爆一拍”的节奏差。
  - **被动**: 白球每次触发时，给场上最近黄球和黑球各挂一个 `押后印记`：黄球的下一次被动会在再下一轮额外补一次低伤追击；黑球的下一次被动会在再下一轮返还一部分冰花或直接补到 4 的整数阈值。
  - **激发**: 立刻引爆所有押后印记，让积压的黄球追击与黑球返还一次性兑现。
  - **优点**: 轮转张力强，适合你的“推进回合 -> 养球 -> 挤出”框架；单白无用，但混搭时会让一轮后的价值明显提高。
  - **风险**: 爆发窗口可能太极端，容易出现“平时偏弱、挤出时过猛”的摆动。
  - **主要平衡旋钮**: 押后印记数量上限严格限制为每种 1 层，避免连锁囤积。

- **初步比较结论**:
  - 若要 **最直观、最好理解**: A 导流白球。
  - 若要 **最鼓励混搭、最反对单色堆叠**: B 共振白球。
  - 若要 **最有原创辨识度、最符合“辅助而非第三输出球”**: C 折射白球。
  - 若要 **最有轮转张力和挤出爆发感**: D 押后白球。

- **当前推荐**:
  - 基于新增约束，当前推荐从单一 `C 折射` 调整为 **A 导流 + D 押后 的混合方向**。
  - 核心原因：
    - `导流` 负责把黑球的防御价值明确导向黄球的进攻价值，直接提升黑球的上场吸引力；
    - `押后` 负责引入非完全确定的兑现时机与兑现形式，避免形成唯一最优固定序列。
  - 这意味着白球更适合被定义为：**把黑球参与产生的“机会值”储存起来，再以有限随机、可预告的形式导给黄球或黑球。**

- **随机性设计原则（新增）**:
  - 若保留随机性，应降到非常轻：只做小幅目标选择偏差或小幅奖励分支，而不是完整的支援池系统。
  - 当前优先级是先保证玩法闭环简单成立，再决定是否补极轻随机。

- **新的推荐方案雏形：助推白球**
  - **核心幻想**: 白球自己不负责产出，而是让“别的球每次触发被动时更值钱”。
  - **被动**:
    - 每次白球被动时，随机强化场上一颗其它球的下次被动。
    - 若强化到黄球：黄球下次被动追加一次低倍率追击，或提高命中高价值目标的稳定性。
    - 若强化到黑球：黑球下次被动额外多给冰花。
    - 如果场上只有黄球没有黑球，白球几乎只是在给黄球做轻度加成；
      但如果场上有黑球，黑球就会因为“更容易产出更多盾 -> 更容易安全受击 -> 更容易推进更多轮被动”而形成连锁收益。
  - **激发**:
    - 被挤出时，立刻让当前场上的其它球各触发一次“强化版被动”，或让其中随机两颗立刻结算强化后的被动。
    - 这样白球激发依然是“放大别人”，而不是自己爆发。
  - **为什么它足够简单**:
    - 只有一句核心规则：`白球强化其它球的被动`。
    - 不需要额外资源条，不需要预告池，不需要复杂状态记忆。
  - **为什么它能抬高黑球价值**:
    - 黑球本来只提供防守；被白球强化后，它会更快叠盾。
    - 盾越多，你就越有资本通过受击来推进更多轮被动。
    - 于是黑球不再只是“少扣血”，而是“帮助整套系统多转几轮”。
  - **为什么它能对抗纯黄球**:
    - 纯黄球只是在每一轮里把单次输出做高。
    - `黄+黑+白` 则是在想办法创造更多“可承受的触发轮数”，再让白球把这些额外轮数里的黄球/黑球被动一起放大。
    - 只要黑球被强化后创造出的额外触发次数足够多，这套组合的总收益就能超过纯黄球。
  - **极简随机性建议**:
    - 随机性只保留在“白球本轮强化哪一颗其它球”。
    - 若场上同时有黄球和黑球，则白球在两者间随机选择；如果有多颗同类球，则再随机选其中一颗。
    - 这样不会出现唯一固定背板，但玩家仍能理解：白球总是在帮别人变强，只是这轮偏黄还是偏黑不完全固定。

- **实现提醒（基于当前代码事实）**:
  - 当前代码里，`OnHeroTookShieldedDamage(...)` 确实会推进回合，但调用 `AdvanceRound(..., triggerIceOrbPassives: false)`，也就是“耗盾受击”时当前实现并不会触发黑球自身被动。
  - 因此如果采用“助推白球”方案，PLAN 阶段需要重点确认：
    - 是保持现状，让黑球只通过“更多已有护盾”间接换来更多受击推进；
    - 还是顺手调整这条链路，让黑球在耗盾受击时也能参与某种有限结算。

- **已选定方向：回响白球（最终草案）**
  - **核心规则**:
    - 白球自己的被动不再直接造成伤害。
    - 白球自己的激发不再直接造成伤害。
    - 白球只负责让其它球的被动/激发被重复结算。
  - **被动规则**:
    - 设当前场上白球数量为 `W`，非白球数量为 `N`。
    - 若 `N == 0`，则白球被动无效果。
    - 每一颗非白球的本轮**额外被动触发次数** = `W`。
    - 因此每一颗非白球的本轮**总被动触发次数** = `1 + W`（原始 1 次 + 白球提供的额外次数）。
    - 因此：
      - `1 白 + 2 其它` → 两颗其它球都各**额外**触发 `1` 次被动，也就是总共各触发 `2` 次。
      - `2 白 + 1 其它` → 这一颗其它球**额外**触发 `2` 次被动，也就是总共触发 `3` 次。
      - `3 白` → 无效果。
    - 这里的“多次触发”按**重复结算同一颗球自己的原始被动**理解，而不是让白球递归调用其它白球。
    - 这里的结算是**顺序无关**的：设计与实现都不应依赖左到右、右到左或其它排序语义，只取决于数量。
  - **激发规则**:
    - 当被挤出的球是白球时，白球激发改为：触发当前其它球的所有激发效果。
    - 设白球离场前场上白球数量为 `W`，非白球数量为 `N`。
    - 若 `N == 0`，则白球激发无效果。
    - 每一颗其它球的**额外激发次数** = `W`。
    - 因此：
      - `1 白 + 2 其它` → 两颗其它球都各额外激发 `1` 次。
      - `2 白 + 1 其它` → 这一颗其它球额外激发 `2` 次。
      - `3 白` → 无效果。
    - 同样，激发规则也应视为**顺序无关，只看数量**。
    - **冰球额外激发补盾细则**:
      - 白球带来的冰球额外激发继续沿用冰球原有激发语义，也就是额外补充冰盾。
      - 这里不需要额外引入“冰盾抵挡伤害窗口”上下文；直接复用冰球原激发逻辑即可。
      - 目标是让“黑球扛住一次 -> 白球放大冰球额外激发 -> 冰盾得到一定返还”成为成立的防御闭环。
      - 该补盾只属于**白球放大的额外激发部分**；冰球自身原始激发仍按原规则执行。
- **设计收益解释**:
    - 白球不把盾直接转成伤害。
    - 它只是让黄球和黑球的原本机制被更频繁地结算。
    - 黑球因此变强，不是因为它变成输出，而是因为它能更稳定地把“额外被动次数”转化成更多可承受回合。
    - 黄球则吃到这些额外轮次里的重复被动，输出自然被抬高。
    - 在防御场景里，白球放大的冰球额外激发还能补回部分冰盾，因此黑球的价值不仅是“多挨几次”，还包括“更容易把一次成功防御滚成下一次防御”。
  - **平衡自限**:
    - 白球自己零直伤、零独立收益。
    - `3 白` 为空场，无自洽闭环。
    - 这保证白球不能单飞，也不会替代黄球或黑球的主职能。
  - **实现建议**:
    - 不要让 `WhiteOrbDefinition.OnPassive()` 或 `OnEvocation()` 直接递归调用其它定义。
    - 正确做法是在 `OrbSystem.TriggerPassiveOrbs(...)` 与 `TriggerEvocation(...)` 层面统一计算重复次数，再按次数循环调用非白球定义。
    - 这样可以避免递归、重复移除、状态错乱和日志难以追踪的问题。
